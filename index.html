01 npm create vite@latest vs npm init Setup

npm create vite@latest downloads and runs the Vite scaffolding tool. It asks questions (project name, framework, TypeScript or JS) and generates a complete, ready-to-code project with folder structure, config files, and sample components. Then you run npm install and you're ready.

npm init is a bare-bones command built into npm itself. It only creates a package.json file — nothing else. You'd then have to manually install React, set up Vite, create all your files, and configure everything from scratch.

💡 For learning React, always use npm create vite@latest — it replaced the older create-react-app that you may see in older tutorials.
02 What build tools actually do Setup

Browsers can't understand JSX or modern JS imports directly. Build tools sit between your code and the browser:

Bundling — stitches all your files into one (or a few) optimized files so the browser doesn't make 50+ network requests.

Transpiling — converts JSX like <App /> into React.createElement(...) that the browser understands.

Dev Server — when you run npm run dev, Vite spins up a local server with Hot Module Replacement (HMR) — only changed parts update instantly without a full page reload.

Production build — npm run build minifies code, removes unused code (tree-shaking), and produces a lean dist/ folder ready to deploy.

💡 Build tools are translators + optimizers. They turn developer-friendly code into browser-friendly, fast, production-ready code.

03 npm run dev & scripts Setup

Yes, npm run dev comes pre-configured by Vite. It's defined in package.json under the scripts section:

"scripts": {
  "dev": "vite",
  "build": "vite build",
  "preview": "vite preview"
}

npm run dev starts the dev server. npm run build creates the production build. npm run preview locally tests your production build.

06 CLI & Bash Concepts

CLI (Command Line Interface) — a way to interact with your computer by typing text commands instead of clicking. The app you type into is called a terminal.

Bash — a specific shell language that runs inside the terminal. It interprets and executes your commands. Most common on Mac/Linux. On Windows the equivalents are Command Prompt and PowerShell. On newer Macs the default is zsh (very similar to bash).

Terminal = the window. Shell (Bash) = the engine inside. CLI = the concept of text-based interaction.

💡 GUI (buttons & clicks) = talking to your computer with a mouse. CLI (text commands) = talking to your computer with words.

Every CLI command follows the same structure:

<tool-name>  <command>  <options>
    npm        install    react
    git         commit   -m "message"

08 Project folder structure Concepts

main.jsx — entry point of your app. First JS file that runs. Mounts your React app into the HTML page. Rarely edited.

App.jsx — root component. Starting point of your actual UI. Where you spend most of your time early on.

App.css — styles specifically for App.jsx.

index.css — global styles for the entire app (fonts, body resets, etc).

assets/ — static files like images and SVGs.

index.html (outside src) — the single HTML file the browser loads. Contains <div id="root"></div> — an empty div that React injects your entire app into.

Browser loads index.html
        ↓
Finds and runs main.jsx
        ↓

        ↓
App.jsx builds the UI from there
09 Root vs child components Concepts

Root component — the top-level component of your app (App.jsx). Everything else lives inside it. There is only one root component in a normal React app.

Child components — everything else you create. Reusable pieces of UI that you plug into your app.

App.jsx  (root)
├── Navbar.jsx
├── Sidebar.jsx
└── MainContent.jsx
    ├── ProductCard.jsx
    ├── ProductCard.jsx
    └── ProductCard.jsx
💡 Think of your UI like LEGO. The root component is the base plate, every other component is a brick. Each brick can hold smaller bricks — that's the component tree.
10 Can there be multiple root components? Concepts

No — there can only be one root component in a React app. main.jsx mounts exactly one component into the #root div.

Technically you can create multiple React roots on one page, but this is only done when gradually migrating an old website to React — not in normal apps.

💡 One React app = one root component (App.jsx) = one component tree. No matter how large your app grows, it all traces back to that single App.jsx.

12 new Xyz vs new Xyz() Concepts

They are identical when no arguments are passed. Parentheses are optional with zero arguments.

Parentheses become required when passing arguments: new Xyz(arg1, arg2)

⚠️ Always write new Xyz() with parentheses — it's more readable, consistent, and avoids subtle operator precedence bugs.
13 JSX curly braces — expressions only JSX

Only expressions (things that return a value) can go inside {} in JSX. Statements (if, for, let) are not allowed.

// ✅ Expressions — allowed
{val1}
{2 + 2}
{isLoggedIn ? "Yes" : "No"}
{getName()}
{count > 0}

// ❌ Statements — not allowed
{if (x > 0) {...}}
{for (let i...) {...}}
{let x = 5}
💡 If you can console.log() it directly, it's an expression and will work in JSX curly braces.

15 Function vs class components Concepts 
// ✅ Function component (modern — use this)
function Header() {
  return <h1>Hello World</h1>
}

// ❌ Class component (old — avoid)

  render() {
    return <h1>Hello World</h1>

Before 2019, class components were needed for state and lifecycle methods. React introduced Hooks in 2019 (useState, useEffect) allowing function components to do everything — with much cleaner code. The whole industry moved to function components.

16 CSS in React — all approaches CSS

1. Global CSS — regular .css files imported in main.jsx. Applies everywhere.

2. CSS Modules — .module.css files scoped to a single component. Most recommended for component styles.

3. Inline Styles — style object directly in JSX. Note double braces and camelCase:

<button style={{ backgroundColor: "blue", color: "white" }}>
  Click
</button>

4. CSS-in-JS — libraries like Styled Components. CSS written inside JS files.

5. Tailwind CSS — most popular in industry right now. Pre-built utility classes applied directly in JSX. No CSS files needed.

Global CSS should be imported in main.jsx — the entry point of the app. This makes it available everywhere, which is exactly what global styles should be.

19 Tag selectors in CSS Modules break scope CSS

CSS Modules scoping only works for class selectors. Tag selectors become global:

/* Button.module.css */
button { background: blue; }  /* ❌ affects ALL buttons in the app */
.btn { background: blue; }    /* ✅ only affects this component */

CSS Modules transforms class names into unique ones (.btn → .btn_x7k2p) but has no way to make tag selectors unique — they stay as-is and apply globally.

💡 Always use class selectors inside .module.css files. Tag selectors belong in index.css (global) only — for intentional app-wide styles like resets.
20 Props Props

Props (properties) are how you pass data from a parent component to a child component.

// Passing props from App.jsx
<Student name="Aman" age={22} interest="React" />

// Receiving props in Student.jsx
function Student(props) {
  return (
    <div>
      <p>Name: {props.name}</p>
      <p>Age: {props.age}</p>
    </div>
  )
}

// Cleaner with destructuring (preferred)
function Student({ name, age, interest }) {
  return <p>{name}</p>   // no props. prefix needed
}

23 JSX elements stored in variables JSX

In JSX you can store entire elements in variables — no quotes needed. Under the hood JSX is just a JavaScript object (React.createElement(...)).

const welcomeMsg = <h2>Welcome</h2>   // ✅ JSX element
const wrongMsg = "<h2>Welcome</h2>"  // ❌ just a string — renders as text literally

// Useful for conditional rendering
const content = isLoggedIn ? welcomeMsg : guestMsg
return <div>{content}</div>
💡 JSX elements are just JavaScript values — store in variables, pass as arguments, return from functions, put in arrays.
24 Conditional rendering Rendering 
// Ternary — if/else (two cases)
{isLoggedIn ? <Dashboard /> : <Login />}

// && — if only (one case)
{isLoggedIn && <Dashboard />}

// ! — reverse a condition
{!isLoggedIn && <Login />}

// || — fallback value
{name || "Anonymous"}

// ?? — fallback only if null/undefined
{name ?? "Anonymous"}

// Return null to render nothing
function Alert({ show }) {
  if (!show) return null
  return <div>Alert!</div>
}

|| vs ??:

💡 Short-circuit = JavaScript stops evaluating as soon as it knows the result. false && anything → stops at false, never evaluates the right side.
26 Rendering lists Rendering

Use .map() to loop over arrays and return JSX — the React way to render lists:

const students = [
  { id: 1, name: "Aman", age: 22 },
  { id: 2, name: "Sara", age: 25 },
]

function App() {
  return (
    <ul>
      {students.map(student => (
        <Student
          key={student.id}
          name={student.name}
          age={student.age}
        />
      ))}
    </ul>
  )
}
27 The key prop Rendering

React uses key to track which list items changed, were added, or removed — so it can update only what changed instead of re-rendering the whole list.

// ✅ Best — unique ID from your data
<Student key={student.id} />

// ✅ Also fine — any unique stable value
<Student key={student.email} />

// ❌ Avoid — array index causes bugs when list reorders
{students.map((student, index) => (
  <Student key={index} />
))}

Key is invisible on the UI — React strips it before rendering. You can't access it inside the component via props either. If you need the ID inside the component, pass it twice:

<Student key={student.id} id={student.id} />
💡 Key must be unique, stable, and not change over time. A database ID is always the best choice.
